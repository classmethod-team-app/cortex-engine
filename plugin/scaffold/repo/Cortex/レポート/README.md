# レポート（日次・週次）

単一案件の「何が起きたか」をまとめたレポートを格納する。**日次**と**週次**の2種を持つ。

## 日次レポート（daily）

「その日のGold昇格ダイジェスト＋概要」。人間（Slack配布）とAI（直近の動きの入口）の両方が当日の動きを1ファイルで掴むための最小レポート。

- 生成: 夜間の `update-gold`（Gold昇格と同一セッションのPhase D。ソースを読んだついでに概要を書くため追加コストは僅少）
- 配置・ファイル名: `records/YYYYMMDD-daily.md`（1日1ファイル・当日再実行で上書き可）
- **動きのない日**: `status: skip` のレコードをワークフローが機械生成する（AI不要）。読み手は frontmatter の `status: skip` を見てボディを読まずに飛ばせる
- 内容: 今日の概要（AI・1〜2段落）＋新しい決定/用語/メンバーの安定ID列挙（機械）＋ソース活動

## 週次レポート（weekly）

**日次レポートの集約＋週の振り返り**。日々の動きは日次が担うため、週次は「週単位の傾向・リスク・来週の計画」という一段上の視点に role を置く。AIS Viewer で顧客・関係者が読む「週ごとの進捗」の顔でもある。

- 生成: `weekly-report` スキル（手動 `/weekly-report` または週次cron `.github/workflows/weekly-report.yml`）。その週の日次レポート（`status: active` のもの）を主要な入力として集約する
- 配置・ファイル名: `records/YYYYMMDD.md`（`YYYYMMDD` は対象週の末日＝生成日。1週1ファイル・再実行で上書き）

## なぜフォーマットを固定するか

このリポジトリは案件ごとに複製される。**全案件が同一フォーマットの週次レポートを持つ**ことで、巡回エージェントが複数案件を横断して機械的に読み取り・集約できる。よって frontmatter と見出しの**並び・文言は固定**とし、案件ごとに独自項目を足さない（該当が無いセクションは削除せず「なし」と記す）。

## 標準フォーマット

雛形は [`template.md`](./template.md)（**これが正本**。frontmatter・見出しともこの構造から変更しない）。

### frontmatterフィールド

| フィールド | 説明 |
| --- | --- |
| type / id | オントロジー規約参照（`id` の `YYYYMMDD` は期間末日） |
| project | プロジェクト名 |
| period_start / period_end | 対象期間（末日の6日前〜末日＝生成日） |
| generated_at | 生成日 |
| metrics | 更新課題数・新規議事録数・新規Decisions数・コミット数・マージPR数 |

### 見出し（この4つ・この順で固定）

| 見出し | 内容 |
| --- | --- |
| 今週の総括 | 週をひとつの物語として3〜5行。重要な決定はここに織り込む（決定の全量列挙は日次と Decisions 一覧が担うため、週次では再掲しない） |
| 進捗と傾向 | 進んだこと／停滞していること。複数の日次を並べて初めて見える兆候（無ければ「特になし」） |
| リスク・懸念・ブロッカー | 停滞・未回答・障害など（無ければ「特になし」） |
| 来週の予定・次アクション | 次にやること・締切・依頼待ち |

数値は本文に書かず frontmatter の `metrics` に一本化する（Viewer・巡回エージェントはそちらを読む）。

## 対象期間

直近7日（生成日を末日とした7日間）。詳細は `weekly-report` スキル参照。
