# Rules

案件で**継続的に守る制約・規約**を構造化して管理するディレクトリ。**1ルール1ファイル**で管理する。

「本番リリースは金曜禁止」「決済処理は必ず二重確認を通す」のような、**その後もずっと効き続ける約束事**を Gold 層のエンティティにする。AI エージェントが作業するとき、この制約を知らずに動くと事故になる（禁止された曜日にデプロイ手順を案内する等）。ルールを機械可読なレコードにしておくことで、巡回エージェント・AIS が案件の「守るべき線」を frontmatter から把握できる。

## Decision との違い

Decision（`Cortex/Decisions/`）と Rule は似ているが役割が違う。

| | Decision | Rule |
| --- | --- | --- |
| 何を記録するか | **経緯のある一回の決定**（いつ・誰が・なぜ決めたか） | **継続的に守る制約**（今も効いている約束事） |
| 時制 | 過去の出来事（凍結・追記型） | 現在も有効な規範（無効化されるまで生きる） |
| 典型 | 「Next.js を採用した」 | 「本番リリースは金曜禁止」 |

Decision から蒸留された Rule（「金曜リリース禁止を決定した」という Decision があり、そこから「金曜リリース禁止」という Rule を立てる）は、`relations` の `derived_from` で元 Decision を指す。

## 命名規則

実データは `records/` 配下に置き、ファイル名は `records/{slug}.md`。ID は `rule:{slug}` となる（例: `rule:本番リリース-金曜禁止`）。

## スキーマ

```yaml
---
type: rule
id: "rule:本番リリース-金曜禁止"
title: "本番リリースは金曜・祝前日に行わない"
description: "障害対応の人員が手薄になる週末直前の本番デプロイを禁止する運用ルール"
status: active          # draft（AI生成・未確認） | active（確認済み）
relations:              # 任意。元になった Decision 等
  - rel: derived_from
    target: "20260610-001"
---

（制約の本文。下の template.md の構成に従う）
```

| フィールド | 必須 | 説明 |
|-----------|------|------|
| type | ✅ | 常に `rule` |
| id | ✅ | `rule:{slug}` |
| title | ✅ | 制約の1行表現 |
| description | ✅ | 制約の1文要約（一覧・検索・外部消費で使う） |
| status | ✅ | `draft`（AI生成・人間未確認）/ `active`（確認済み） |
| relations | - | 元 Decision を `derived_from` で、関連ルールを `relates_to` で指す（任意） |

## 運用（事後レビュー方式）

用語集（term）と同じ**事後レビュー方式**で運用する。

- **AI が起票したルールは必ず `status: draft`**（人間未確認の印）。レビューで内容を確認し `active` に変える
- 自動化の範囲は「新規追加」のみ。**既存レコードの自動書き換えは行わない**
- 効かなくなったルールは削除せず、置き換える Decision から履歴をたどれるようにする（本文に失効を追記するか、レビューで扱いを決める）
- push 前にスキーマ検証（エンジンの `validate-cortex`）が走るため、規約違反のレコードが main に入ることはない

> 現時点では夜間の自動抽出（update-gold へのフェーズ追加）は行わない。ルールは人間または明示的なスキル操作で起票する。
