# メンバー

プロジェクト関係者（開発側・顧客・ベンダー）を構造化して管理するディレクトリ。**1人1ファイル**で管理する。

文字起こしや議事録では、同じ人物が複数の表記（誤変換・下の名前・愛称）で現れると、AIは担当を取り違えたまま自信を持って議事録・Decision を書いてしまう。名簿は**氏名の正式表記と表記揺れを1件にまとめる**ことでこれを防ぎ、**人名正規化の唯一の根拠**になる（用語集が用語の揺れを束ねるのと同じ役割）。ルール層のメンバー表は使わず、この名簿を正本とする。

## 命名規則

実データは `records/` 配下に置き、ファイル名は `records/{氏名（スペース無し）}.md`。IDは `member:{氏名（スペース無し）}` となる（例: `member:山田太郎`）。

## スキーマ

```yaml
---
type: member
id: "member:山田太郎"
title: "山田 太郎"
yomi: "やまだ たろう"
aliases: ["山田さん", "ヤマダ"]
org: "株式会社サンプル"
side: cm
role: "PM"
email: "taro.yamada@example.com"
status: active
---

（本文は任意。担当領域の補足など）
```

| フィールド | 必須 | 説明 |
|-----------|------|------|
| type | ✅ | 常に `member` |
| id | ✅ | `member:{氏名（スペース無し）}` |
| title | ✅ | 氏名の正式表記（表示・正規化のターゲット。スペースを含んでよい） |
| yomi | - | 読み仮名（同姓の区別・音声認識の突合に使う） |
| aliases | - | 文字起こしの誤変換・呼称の揺れのリスト。**新しい表記を見つけたらここに追記する**（用語集の synonyms と同役割） |
| org | - | 所属組織名（開発側／顧客／ベンダー等の正式社名） |
| side | - | `cm`（開発側）/ `client`（顧客）/ `vendor`（ベンダー）。担当割当の整合チェックに使う controlled vocabulary。空も可 |
| role | - | 役割（PM / エンジニア / プロジェクトオーナー 等） |
| email | - | 任意。会議参加者（カレンダー/Meet）との突合に使う |
| status | ✅ | `active`（在籍）/ `inactive`（離任）/ `draft`（AI生成・人間未確認） |

## 内部情報は書かない（顧客開放の境界）

本リポジトリは顧客が直接アクセスする前提で運用する。名簿には**氏名・所属・役割・表記揺れだけ**を書き、**人の評価・人事・単価・工数等の内部情報は絶対に書かない**。判断に迷う情報はそもそも置かないこと。

## 運用

- **正本は人間が管理する**。追加・更新は人間が行い、離任しても削除せず `status: inactive` にする（過去の議事録・Decision の正規化根拠として残す）
- 同一人物に複数ファイルを作らない。表記揺れは `aliases` に集約する
- 定義（氏名・所属・役割）が変わったら本文・frontmatter を**直接更新**する（変更履歴は git が持つ）

## 自動追加（提案のみ・事後レビュー方式）

議事録生成（`create-minute` / `ingest-minutes`）で名簿に無い参加者が現れたとき、AI は **`status: draft` での追加提案のみ**を行う（用語集の自動追記と同じ事後レビュー方式）。

- **AI が追加したレコードは必ず `status: draft`**（人間未確認の印）。レビューで氏名・所属・side を確認し `active` に変える
- 自動化の範囲は「新規追加の提案」のみ。**既存レコードの自動書き換えは行わない**
- push 前にスキーマ検証（エンジンの `validate-cortex`）が走るため、規約違反のレコードが main に入ることはない
