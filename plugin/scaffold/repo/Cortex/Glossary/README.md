# 用語集

案件固有の用語・定義を構造化して管理するディレクトリ。**1用語1ファイル**で管理する。

同じ概念が複数の表記（例: 「コンテキスト」「案件コンテキスト」「context」）で呼ばれると、AIは矛盾を指摘せずどれか1つの解釈を選んで自信を持って誤答する。用語集は**同義語・表記揺れを1つの定義に束ねる**ことでこれを防ぐ。

## 命名規則

実データは `records/` 配下に置き、ファイル名は `records/{代表表記}.md`。IDは `term:{代表表記}` となる。

## 記入例

[`records/コンテキスト.md`](./records/コンテキスト.md) を参照。全案件共通の組織語彙（`scope: organization`）としてテンプレートに同梱されており、複製先でもそのまま用語集の1件目として機能する。

## スキーマ

```yaml
---
type: term
id: "term:コンテキスト"
title: "コンテキスト"
synonyms: ["案件コンテキスト", "context"]
scope: project          # project=案件固有 | organization=全社共通
status: active          # draft（AI生成・未確認） | active | superseded（使われなくなった用語）
date: 2026-06-10        # 登録日
source: "https://..."   # この案件での出典（Backlogドキュメント・議事録の安定ID等。任意）
references:              # 一般公開用語の一次情報リンク（公式ドキュメント等。案件固有語は省略）
  - "https://www.figma.com/developers/api"
---

（定義本文。1〜数段落で、この案件における意味を具体的に書く）
```

| フィールド | 必須 | 説明 |
|-----------|------|------|
| type | ✅ | 常に `term` |
| id | ✅ | `term:{代表表記}` |
| title | ✅ | 代表表記 |
| synonyms | - | 同義語・表記揺れのリスト。**新しい表記を見つけたらファイルを増やさずここに追記する** |
| scope | ✅ | `project`（案件固有）/ `organization`（全社共通） |
| status | ✅ | `draft`（AI生成・人間未確認）/ `active`（確認済み）/ `superseded` |
| date | ✅ | 登録日 |
| source | - | **この案件での出典**（どこで使われていたか。Backlogドキュメント・議事録の安定ID/URL等） |
| references | - | **一般公開用語の一次情報リンク**（公式ドキュメント等のURLリスト）。案件固有語は省略 |

### 定義の正確さと参考リンク（一般公開用語）

用語は「案件固有語」と「一般に公開されているサービス・技術・標準語」に分かれる。後者は推測で書かず、**Web検索で公式・一次情報を確認**してから定義する。

- **案件固有語・業界内部語**（その案件/社内でしか出てこない語）: 社内の情報源（議事録・Backlog等）の定義のみを使う。`source` に案件内の出典を書き、`references` は付けない。Web検索しない（公開情報が無い／案件文脈が正）。
- **一般公開のサービス・技術・標準語**（例: Figma / LINEミニアプリ / OIDC / Backlog 等）: **Web検索で公式ドキュメント等の一次情報を確認**し、正確な定義を書く。`references` に**権威ある一次ソースのURL**を1〜数件記載する。一般的な意味に加えて、この案件での使われ方が特殊なら本文でその点も補足する。
- `source`（案件内の出典）と `references`（外部の権威リンク）は役割が違うので両立する。

## 運用

- 追加・更新は `/update-glossary` スキル経由で行う（AI抽出 → 人間承認）
- 定義が変わったら本文を**直接更新**する（変更履歴はgitが持つ。Decisionsと異なり追記型ではない）
- 使われなくなった用語は削除せず `status: superseded` にする
- 同一概念に複数ファイルを作らない。表記揺れは `synonyms` に集約する

## 自動更新（直コミット・事後レビュー方式）

毎晩、`update-glossary-auto` スキル（Gold昇格統合ワークフロー `.github/workflows/update-gold.yml` の用語フェーズ）がその日の差分から新規用語を抽出し、**`status: draft` でmainに直接コミット**する。承認はPRマージではなく**事後レビュー**で行う。

- **AI生成の用語は必ず `status: draft`**（人間未確認の印）。ViewerはdraftのレコードにAI生成・未確認の注意を表示する
- **レビュー = 定義を確認・修正して `status: active` に変える**こと。AIが生成した定義は仮のものとして扱い、必ず人間が内容を確認して適切な定義に置き換える
- 自動化の範囲は「新規追加」と「synonyms追記」のみ。**既存定義の自動書き換えは行わない**（更新型のGold層のため、誤った定義のドリフトを構造的に防ぐ）
- 不要な用語は、該当ファイルを削除し、下記「除外用語」に語を追記すると再追加されなくなる
- push前にスキーマ検証（エンジンの `validate-cortex`）が走るため、規約違反のレコードがmainに入ることはない

## 除外用語

自動更新（`update-glossary-auto`）が**追加してはいけない語**のリスト。レビューで却下した語をここに追記すると、以後再追加されなくなる（編集は人間のみ。自動更新はこのリストを読むだけで編集しない）。
