# デザイン（画面育成ノート）

デザイン画面の**育成ノート**を Gold 層のエンティティとして管理するディレクトリ。**1画面1ファイル**で、`records/` 配下に置く。

デザインには2つの層がある。混同しないこと。

| 層 | 実体 | 誰が書く |
| --- | --- | --- |
| **Silver（目録）** | `デザイン/inventory/`（画面一覧・安定ID・Figmaディープリンク・サムネイル・機械抽出テキスト） | `sync-designs` が Figma から**同期ミラー**（毎回全再生成・手編集禁止） |
| **Gold（育成ノート）** | `Cortex/デザイン/records/`（画面の目的・主要要素・関連する用語/Decision） | AI が視覚AIで draft 生成 → 人間がレビューして active（このディレクトリ） |

同じ画面の Silver と Gold は同じ安定ID `design:{fileKey}:{nodeId}` を共有する。目録は同期で毎回作り替わるため蓄積に向かない。画面の意味・意図・文脈の**蓄積は育成ノート（Gold）側**が担う。

## 命名規則

実データは `records/` 配下に置き、ファイル名は画面名を safe 化し **nodeId を末尾に含めて衝突を避ける**（例: `records/ログイン画面-1023-456.md`）。ID は `design:{fileKey}:{nodeId}`。

## スキーマ

```yaml
---
type: design
id: "design:{fileKey}:{nodeId}"
title: "{画面名}"
status: draft   # draft(AI生成・未確認) | active(人間確認済み)
source: "デザイン/inventory/ の該当画面（正本はFigma）"
relations: []   # 関連する用語・Decision等（rel: relates_to で安定ID）
---
```

| フィールド | 必須 | 説明 |
|-----------|------|------|
| type | ✅ | 常に `design` |
| id | ✅ | `design:{fileKey}:{nodeId}`（inventory の参照IDと一致） |
| title | ✅ | 画面名 |
| status | ✅ | `draft`（AI生成・未確認）/ `active`（人間確認済み） |
| source | - | inventory の該当画面（正本は Figma） |
| relations | - | 関連する用語・Decision 等（`rel: relates_to` で安定ID） |

## マーカーによる AI 領域と人間領域の分離

本文は `<!-- cortex-auto:begin -->`〜`<!-- cortex-auto:end -->` の**マーカーで区切る**。

- **マーカーの内側は AI が再生成する領域**（画面の目的・主要要素・主要テキスト・Figmaディープリンク・関連リンク）。**人間はここを編集しない**（次回の自動更新で上書きされる）。
- **人間の補足はマーカーの外**（`## 補足` 以下）に書く。AI はマーカー外を一切変更しない。
- 画面が Figma 側で大きく変わると、AI が auto 領域を更新し、`status` を `draft` に戻す（人間の再確認を促す）。

## 運用（draft 事後レビュー方式）

用語集・メンバーと同じ**事後レビュー方式**。

- AI が視覚AI（サムネイルPNGの読み取り）で auto 領域を生成し、必ず `status: draft` を付ける
- 人間が内容を確認・修正し、`status: active` に変えてレビュー完了とする
- 自動化の範囲は「auto 領域の生成・更新」のみ。**マーカー外（人間の補足）と、既存レコードの人間確認済み定義は自動で書き換えない**
- push 前にスキーマ検証（エンジンの `validate-cortex`）が走るため、規約違反のレコードが main に入ることはない

## 内部情報は書かない（顧客開放の境界）

本リポジトリは顧客が直接アクセスする前提で運用する。育成ノートには**画面から読み取れる事実・意図の要約だけ**を書き、**画像から読み取れないことを断定しない**（幻覚防止）。内部限定情報（評価・単価・工数等）は絶対に書かない。
