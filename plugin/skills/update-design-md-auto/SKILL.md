デザイン版CLAUDE.mdである `$DESIGN_DIR/DESIGN.md`（1ファイル）の**本文（公式8節のprose）**を自律育成するスキルです。前回実行以降に `デザイン/inventory/`・`デザイン/resources/` が変更されたとき、Figmaから取れる事実に基づいて `{{例: …}}` プレースホルダを実文に置き換え、既存の記述を最新のデザインと整合させます（push はワークフロー側の責務）。

## 設計の要点

- **2層の分業を厳守する**: YAMLフロントマター＝デザイントークン（sync-designs の機械領域）／Markdown本文＝設計判断・ガードレール（このスキルと人間の領域）。**フロントマターは1文字も変更しない**
- **根拠にできる素材は次の3つだけ**（これ以外から断定しない＝幻覚防止）:
  1. DESIGN.md フロントマターのトークン値（色・タイポ・角丸・スペーシング）
  2. inventory md の機械抽出節（`## 画面内テキスト（機械抽出）`・`## 使用コンポーネント（機械抽出）`）
  3. サムネイルPNG（`$DESIGN_DIR/resources/`）を **Read（画像）** で見る。**代表画面 最大10枚まで**（直近変更された画面を優先）
- **人間の実文を尊重する**: 既に書かれた実文は、現在のトークン・画面と矛盾した場合（例: 色変更・フォント変更）だけ更新する。それ以外は削らない・言い換えない。迷ったら更新せず残す（保守側に倒す）
- **公開範囲**: 顧客が直接見る前提。内部限定情報（評価・単価・工数等）は書かない

## 実行手順

### ステップ 1: 変更の把握と素材の収集

環境変数 `DESIGN_DIR`（既定は `デザイン`）・`SINCE`（前回成功run時刻。ワークフローが渡す。手元実行では約25時間をデフォルト）を使う。

```bash
DESIGN_DIR="${DESIGN_DIR:-デザイン}"
git -c core.quotepath=false log --since="${SINCE:-25 hours ago}" --name-only --pretty=format: \
  -- "$DESIGN_DIR/inventory/" "$DESIGN_DIR/resources/" | grep -v '^$' | sort -u
```

1. `$DESIGN_DIR/DESIGN.md` を読む。フロントマターのトークン値（colors / typography / rounded / spacing）と、本文8節の現状（プレースホルダのまま or 実文あり）を把握する
2. 変更された inventory md を読み、機械抽出節（画面内テキスト・使用コンポーネント）を集める。変更が広範囲なら inventory 全体をざっと見て全体像（画面数・画面の種類）を掴んでよい
3. サムネイルPNGを Read（画像）で見る。**直近変更された画面を優先して最大10枚まで**。`safeNodeId` は nodeId の `:` を `-` に置換したもの（inventory と同じ規則）

### ステップ 2: 本文8節の記入・更新

公式8節（Overview / Colors / Typography / Layout / Elevation & Depth / Shapes / Components / Do's and Don'ts）の**順序・見出しを維持**して編集する。節の追加・削除・並べ替えはしない。

- **`{{例: …}}` プレースホルダ**は、根拠のある実文に置き換える。**根拠が足りない箇所はプレースホルダのまま残す**（無理に埋めない）
- **既に書かれた実文**（人間または過去のAIによる）は尊重する: 現在のトークン・画面と矛盾した場合だけ更新し、それ以外は削らない・言い換えない。編集は追記・修正最小限
- **抽象語（プレミアム・モダン等）で書かない**。具体値と理由で書く（例: 色はフロントマターのトークン名とhex値を挙げ、サムネイル・機械抽出から観察できる使われ方を書く）
- 各節に書く内容の目安:
  - **Overview**: 画面群から観察できる密度・トーン（根拠が薄ければプレースホルダのまま）
  - **Colors / Typography / Layout / Shapes**: フロントマターのトークン値を prose で意味付けする（`- **Primary (#hex):** 用途` の形式。用途はサムネイル・機械抽出で観察できた範囲で）
  - **Elevation & Depth**: サムネイルから観察できる階層表現（影か面か）。観察できなければ残す
  - **Components**: 使用コンポーネント（機械抽出）とサムネイルから観察できるものだけ
  - **Do's and Don'ts**: 観察された一貫性（例: 角丸の統一）をガードレール文に。推測で禁止則を作らない
- **フロントマター（先頭の `---`〜`---`）は絶対に変更しない**。sync-designs が毎晩再生成する機械領域

### ステップ 3: 検証・コミット

1. **公式lintを実行**し、errors があれば修正する（warningは許容）:
   ```bash
   npx -y @google/design.md@0.3.0 lint "$DESIGN_DIR/DESIGN.md"
   ```
2. `git add "$DESIGN_DIR/DESIGN.md"`（**このファイルのみ**）
3. 変更があれば `git commit` までを行う（push はワークフロー側）。コミットメッセージは日本語で簡潔に、箇条書きは使わず、AI署名は付けない
4. 変更が無ければ何もコミットせず正常終了

## 注意事項

- **フロントマターを1文字も変更しない**（sync-designs の機械領域。機械領域と人間領域の分業を壊すのが最悪の事故）
- **根拠素材（トークン・機械抽出・サムネイル最大10枚）以外から断定しない**（幻覚防止）
- **迷ったら更新せず残す**（既存の実文の扱いは保守側に倒す）
- **内部限定情報は書かない**（顧客開放の境界）
- inventory・resources（Silverミラー）は**読むだけで編集しない**（正本はFigma・sync-designsが再生成する）
- `Cortex/` 配下・`.cortex-engine/` 配下は編集しない・編集源として自己参照しない
