**前回実行以降に追加・更新されたコンテキスト**（その日の議事録・課題など）を走査して確定した意思決定を抽出し、`Cortex/Decisions/records/` 配下に **1決定1ファイル**で**重複なく追記**します。

`update-decision-log`（手動・承認あり）の**自律実行版（増分方式）**です。担当者が叩かなくても、毎晩 cron がその日の決定を自動で積み上げます。

## 設計の要点

- **1決定1ファイル**: `Cortex/Decisions/template.md` をベースに、`Cortex/Decisions/records/YYYYMMDD-NNN-決定内容の要約.md` を新規作成する（既存ファイルは絶対に変更・削除しない）。形式・フィールドは `Cortex/Decisions/README.md` に従う。
- **増分方式**: 毎晩フル走査せず、前回実行以降に追加・更新されたソースだけを対象にする。会議は毎日あるわけではないので、**新規が無い日は「追記なし」で正常終了**するのが通常。
- **取りこぼし防止**: 「会議日フォルダ == 今日」ではなく **git の差分（前回実行以降に追加・更新されたファイル）** を対象にする。議事録が会議翌日に生成されてもズレなく拾える。
- **重複防止は保険**: 対象を当日差分に絞るため重複は起きにくいが、実行窓を少し重ねる（約 25 時間）ため、既存 Decision との照合で念のため二重登録を防ぐ。

## 実行手順

### ステップ 0: 課題の最新化（任意）

Backlog 由来の当日の決定を取りこぼさないよう、最初に課題を最新化する。失敗してもログを残して続行する。

```bash
pnpm dlx backlog-exporter@1 update --force || echo "backlog同期に失敗。既存ローカルファイルで続行します"
```

> cron ワークフロー側で既に実行済みの場合はこのステップをスキップする。

### ステップ 1: その日の差分ソースを特定

**前回成功実行以降**（環境変数 `SINCE`。ワークフローが「直近の成功run時刻」を渡す＝失敗が続いても次の成功時に失敗分まで遡って巻き取る）に追加・更新された、意思決定が含まれるファイルだけを対象にする。`SINCE` が無い手元実行では約 25 時間をデフォルトにする。`Cortex/Decisions/` 配下は必ず除外する（自己参照ループ防止）。重複は署名照合（ステップ4）で吸収するので、窓が多少重なっても二重登録にはならない。

```bash
# SINCE（前回成功以降）にコミットで追加・更新されたファイルを対象にする。SINCE未設定なら約25時間。
# ★ソース（Bronze/Silver）のディレクトリ名は案件ごとに任意（MTG/・チャットエクスポート・想定外の名前が入る）ため、
#   特定ディレクトリに絞らず「コンテキストに入ってきたもの全部」を対象にする。除外はエンジン（.cortex-engine/）と
#   自分たちの生成物であるGold層（Cortex/。決定を決定から抽出する自己参照ループを防ぐ）だけ。
#   さらに日本語パス（会議/・課題管理/・MTG/運用保守/ 等）が8進エスケープされ .md$ を取りこぼすのを防ぐため
#   core.quotepath=false を付ける（案件のソースはほぼ日本語ディレクトリなので必須）。この判定ロジックは
#   下記スクリプトに集約している（コミット済み差分＋未コミットの作業ツリー変更の和集合を1行1件で出力）。
# ワークフローの差分ゲートと同一スクリプトで判定する（二重定義によるドリフト防止）
bash "<SKILL_DIR>/../../scripts/changed-sources.sh" "${SINCE:-}" "Cortex/"
```

- 出力された各行を「当日の対象ソース」とする。
- **適応的検出**: このリポジトリは案件ごとに複製される。意思決定が記録される新規ディレクトリ（Slack/Teams/チャットのエクスポート等）が増えていれば、同スクリプトが自動で差分対象に含める（パスをハードコードしない）。
- 対象が 0 件なら、以降をスキップして「追記なし」で正常終了する。ワークフロー経由の実行では、このコマンドが0件ならそもそもAI実行前にスキップされている。手元実行（/コマンド）ではこの結果が0件なら以降をスキップして正常終了する。

### ステップ 2: 採番と重複照合の準備（既存 Decision は全文読みしない）

> ⚠️ **既存 Decision を 1 ファイルずつ開いて全文 Read しない**。records は案件の進行とともに数百件規模になり、全件 Read は無人実行のターン上限を食い潰す。ここで必要なのは「採番用のファイル名」と「重複照合用の frontmatter（`title`/`description`）」だけで、**どちらもコマンド数回で取れる**。

- **日付ごとの最大連番**（`YYYYMMDD-NNN` の `NNN`。次に振る番号の起点）は **ファイル名だけで分かる**（中身は不要）。これから書く決定の日付について最大連番を取る：

```bash
# 例: 2026-06-26 の決定を書くなら、その日付の既存最大連番を見る（無ければ 000 扱い→001 から振る）
ls Cortex/Decisions/records/ | grep -E '^20260626-' \
  | sed -E 's/^[0-9]{8}-([0-9]{3}).*/\1/' | sort -n | tail -1 || true
```

- 既収録レコードの**署名集合**（重複判定用。frontmatter `title` ＋ `description` を空白・記号正規化した文字列）は、**frontmatter だけを grep で一括取得**する（ファイルを開かない）。増分窓は約 25 時間で、抽出される決定の日付は直近に限られるため、照合相手は「**これから書く日付の既存 Decision だけ**」で十分（古い決定とは衝突しない）：

```bash
# これから書く日付ぶんだけ、id/title/description を 1 回で取得（全文 Read しない）
grep -hE '^(id|title|description):' Cortex/Decisions/records/20260626-*.md 2>/dev/null || true
```

> 対象日付が先に分からなければ、ステップ 3 で決定を抽出して日付を確定させてから本ステップを行ってもよい（順序は前後して構わない）。どうしても全体の署名が要る場合でも、ファイルを開かず `grep -hE '^(title|description):' Cortex/Decisions/records/*.md` の **1 コマンド**で済ませる。

### ステップ 3: 当日分の確定した決定事項を抽出

ステップ 1 で特定した対象ソースからのみ、確定済みの決定を抽出する。

**抽出対象（確定表現の例）**：

- 「〜で決定」「〜にした」「〜で進める」「〜で合意した」「〜方針とする」等の**完了・確定**の表現
- 議事録は「決定事項まとめ」セクションを主たる情報源とする
- 課題は、質問 → 回答で方針が確定したやり取り・合意表現
- Slack等のチャット連携があれば、確定した方針のスレッド

**除外するもの**：

- 質問のみ・回答待ち（未確定）、「確認中」「検討中」等の未確定表現
- 「〜で合意する」「〜で決める」「後段で合意」「本会で確定」等、**これから決める予定**を表す未来形・予定表現（=アジェンダ論点。確定したらその定例の議事録から拾う）
- アカウントセットアップ・環境構築等の運用作業

**⚠️ 定例準備ファイル（アジェンダ）は決定の抽出源にしない**：

未開催の定例ファイルは協議予定の論点が並ぶだけで決定はまだ存在しない。差分に現れても、`## アジェンダ` のような協議予定セクションや、`決定事項まとめ` が空（未記入）／`想定実施日` が未来日のファイルからは抽出しない。開催後に `決定事項まとめ` が埋まったら通常どおり抽出する。

**抽出範囲の方針**：機能要件に限らず、仕様・設計・運用・ビジネスの確定事項すべてを対象とする（カテゴリーは `Cortex/Decisions/README.md` の一覧から選ぶ）。

### ステップ 4: 重複照合 → 新規のみに絞る

抽出した各決定をステップ 2 の署名集合と照合し、署名一致または**意味的に同一**のものは破棄。残った真に新規の確定事項のみを追記対象とする（実行窓のオーバーラップ分はここで吸収される）。新規が 0 件なら以降をスキップして正常終了。

### ステップ 5: Decisionファイルの作成

新規の決定ごとに、`Cortex/Decisions/template.md` をベースに1ファイルずつ作成する。

- **ファイル名**: `Cortex/Decisions/records/YYYYMMDD-NNN-決定内容の要約.md`
  - `YYYYMMDD` は**決定が行われた日**（会議日など。実行日ではない）
  - `NNN` は同日内の連番（ステップ 2 の最大連番 +1 から）
- **frontmatter**（詳細は `Cortex/Decisions/README.md`）:
  - `type`: `decision`（固定）
  - `id`: `YYYYMMDD-NNN`
  - `title` / `date` / `category`（README のカテゴリー一覧から）/ `deciders` / `description`
    - `deciders` は名簿（`Cortex/Members/records/` の `title`）の正式表記に正規化する。名簿に無い人名は推測で確定させず `（要確認）` を付けて記載する
  - `sprint`: スプリント運用している案件のみ記入（無ければ省略可）
  - `relations`: 抽出元への型付き関係を安定IDで記載（議事録: `rel: based_on, target: "minute:{定例名}:{YYYYMMDD}"`、課題: `rel: based_on, target: "{課題キー}"`。過去の決定の変更なら `rel: supersedes, target: "{決定ID}"`）。**ファイルパスは使わない**
  - `references`: 抽出元（議事録パス・課題キー＋コメント・外部リンク等）を必ず記載（人間向けの併記）
- **本文**: `## 背景` と `## 理由` を記述

### ステップ 6: 直コミット（人手承認なし）

```bash
git add Cortex/Decisions/
if git diff --staged --quiet; then
  echo "追記なし"
else
  git commit -m "Decisionsに当日の決定事項を自動追記"
fi
```

- **コミット前に `node "<SKILL_DIR>/../../scripts/validate-cortex.mjs"`（プラグイン同梱リンター） を実行し、規約違反があれば修正してからコミットする**（違反したままコミットしない）
- コミットメッセージは**日本語で簡潔**に。箇条書きは使わない。**AI 署名は付けない**。
- push は呼び出し側（cron ワークフロー）が担当する。

## 注意事項

- 増分方式（その日の差分のみ）。過去分の一括取り込みは行わない（既存の積み残しは別途バックフィルで対応）。
- 既存の Decisionファイルは絶対に変更・削除しない（新規作成のみ）。
- 既存 Decision は **採番＝ファイル名・重複照合＝frontmatter の grep** で扱い、**1 ファイルずつ全文 Read しない**（件数増でターン超過するため。ステップ 2 参照）。
- 未確定・検討中の議論は追加しない。確定した決定のみ。
- `Cortex/Decisions/` 配下は探索対象から除外する（自己参照防止）。
