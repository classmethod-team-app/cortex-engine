コンテキストリポジトリ全体を見渡して、プロジェクトの最新状況を整理します。直近の定例会、課題管理ツールの動き、Decision Log、PBI の状況などから、今何が起きていて何が論点になっているかを把握します。

## 実行手順

### ステップ 1: 直近の定例会議事録を確認

`会議/` 配下から直近 2 回分の定例会議事録（`*_minutes.md`）を読み込む。

確認ポイント：

- **決定事項**: 何が決まったか
- **TODO アイテム**: 誰が何を担当しているか、進捗はどうか
- **議論中の論点**: まだ決まっていないこと、次回に持ち越したこと

### ステップ 2: 課題管理ツールの最近の動きを確認

`課題管理/issues/` 配下の課題ファイルから、最近のコメント（直近 1 週間程度）を検索する。

確認ポイント：

- **新規作成された課題**: 新たに発生した要件や確認事項
- **回答があった課題**: 外部ベンダーやクライアントからの回答
- **進捗があった課題**: ステータスが変わったもの
- **未回答の課題**: 投げかけたまま返答待ちのもの

### ステップ 3: Decision Log の最新レコードを確認

`Cortex/Decisions/` 配下のファイルを日付（ファイル名の `YYYYMMDD`）の新しい順に、最新のレコード（直近 10 件程度）を確認する。

確認ポイント：

- **最近の決定事項**: 何がいつ決まったか
- **仮おきの決定**: 確認中など、まだ確定していない決定

### ステップ 4: 要件定義書の状況を確認

`課題管理/documents/` 配下の要件定義書等を確認する。

確認ポイント：

- **ステータス**: 各機能が「確定」か「検討中」か
- **残っている検討事項**: 未解決の質問や確認事項

### ステップ 5: PBI・GitHub Issue の状況を確認

`開発/issues/` 配下の PBI ファイルを確認し、GitHub 上の Issue 状況も確認する。

```bash
gh issue list --repo {{開発リポ}} --limit 20
```

### ステップ 6: 外部関係者の対応状況を確認

クライアントや外部ベンダーへの依頼事項の進捗を、課題管理や議事録から確認する。

確認ポイント：

- **依頼中の事項**: 何をいつ誰に依頼したか
- **回答待ちの事項**: 返答が来ていないもの
- **確認済みの事項**: 回答が得られたもの

### ステップ 7: レポート出力

以下の形式で整理して報告する。

```markdown
## プロジェクト最新状況（YYYY/MM/DD 時点）

### 直近の主な動き

- [箇条書きで 3-5 個]

### 現在の論点・検討中の事項

- [未決定の事項、議論中のテーマ]

### 外部関係者の対応状況

| 関係者 | 対応事項 | ステータス |
| ------ | -------- | ---------- |

### 次のアクション

- [今週やるべきこと、次回定例で確認すべきこと]

### 進捗サマリー

- 要件定義: [状況]
- PBI: [件数・状況]
- 開発: [フェーズ]
```

## 注意事項

- **最新の日付を基準にする**: ファイルの更新日時やコメントの日時で鮮度を判断する
- **古い情報に惑わされない**: 同じ論点で複数の情報がある場合、最新のものを優先する
- **技術用語は最小限に**: レポートは非エンジニアも読む前提で書く
