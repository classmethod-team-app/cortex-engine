description: 会議の文字起こしから構造化された議事録を自動生成する
---
ユーザーから要求された「$ARGUMENTS」の文字起こしファイルから構造化された議事録を生成してください。

## 対応する文字起こし形式

- **Google Meet**: `.txt` ファイル（タイムスタンプ + 発言者: 発言内容）
- **Teams**: `.docx` ファイル（document-skills の docx スキルで読み込み）
- **その他**: プレーンテキストの文字起こし

## ディレクトリ構成の規約

```
会議/
└── Ph.1/
    └── {定例名}/                  # 定例名の階層（例: 全体定例）。単発会議等で定例名が無い場合は省略可
        └── YYYYMMDD/
            ├── YYYYMMDD_minutes.md   # 議事録ファイル（生成対象）
            └── {元の文字起こしファイル} # 文字起こしファイル（入力）
```

- 既存の `会議/` 配下にどの階層構成で議事録が置かれているかを必ず確認し、それに合わせる（自動取り込みされた文字起こしが別の場所にある場合は、規約の場所へ移動してから議事録を生成する）

## 手順

1. **直近の状況を把握する（精度向上の肝・スキップ禁止）** — まず `catch-up-recent-status` で直近の定例会・課題管理・Decision Log を確認し、最近の論点・進行中の議論・前回からの持ち越し TODO を頭に入れる。これを踏まえると、文字起こし中の略語や継続トピックを正しく解釈でき、議事録の精度が上がる。
2. **Gold層でプロジェクト理解と語彙を揃える** — `Cortex/Home.md`（全体像）と `Cortex/Glossary/`（正式表記と synonyms。文字起こしの誤変換を正規化する根拠になる）を読む。会議で「前回決めた〜」と参照されそうな直近の `Cortex/Decisions/` も確認する。
3. **メンバー・体制を確認する** — 名簿（`Cortex/Members/records/`。frontmatter の title/yomi/aliases/side）や体制系 Decision、過去議事録の参加者欄を読み、文字起こしに登場する人名（誤変換・下の名前・愛称を含む）を正しい氏名に対応付ける。文字起こし冒頭の参加者リストとも突合する。名簿が無い案件では、従来どおりメンバー表ルールや CLAUDE.md の表にフォールバックする。
4. **正規化マッピングを作る** — 手順1〜3をもとに「文字起こし上の表記 → 正式表記」の対応表を作ってから執筆に入る（例: 音声認識がプロダクト名をカタカナに誤変換したもの → 用語集の正式表記）。対応が確定できない語・人名は推測で確定させず、`（要確認）` を付けて残す。
5. 指定された／取り込み済みの文字起こしファイルを読み込む（大きい場合は offset/limit で分割読み込み）
6. 正規化マッピングを適用しながら、直近の状況を踏まえて文字起こしから議事録を生成し、同じディレクトリに `YYYYMMDD_minutes.md` として配置する

> 手順1〜4を飛ばして文字起こしをそのまま構造化すると、誤変換された用語・人名や文脈の誤解釈がそのまま Silver 層（議事録）に固定化され、下流の Decision Log・用語集・レポートまで汚染される。必ずプロジェクトを理解してから書くこと。

## 議事録生成ルール

### frontmatterは付けない（IDはパス規約で決まる）

議事録は Silver 層なので frontmatter を付けない（frontmatter を持つのは Gold 層＝`Cortex/` 配下のみ。オントロジー規約参照）。

Gold 層（Decisions 等）からこの議事録を参照するときの安定ID `minute:{定例名}:{YYYYMMDD}` は、ディレクトリ規約（`会議/…/{定例名}/{YYYYMMDD}/YYYYMMDD_minutes.md`）から機械的に導出される。**IDを成立させるため、配置は必ず規約に従う**こと（定例階層がない単発会議は、議題から簡潔な会議名を付けてその名前のディレクトリを作る）。

自動実行（夜間の ingest-minutes 等）で生成した場合は、人間レビュー前であることを本文の H1 直後に1行で明示する（人が内容を確認したらこの行を削除する運用）:

```markdown
> ⚠️ この議事録はAIによる自動生成です（人間レビュー前）。内容を確認したらこの行を削除してください。
```

### 基本構造

`会議/templates/minute-template.md` を参照してテンプレートに従って作成する。

```markdown
# 【日付】会議議事録

## 会議情報

- **日時**: YYYY年MM月DD日（曜日） HH:MM～HH:MM
- **議題**: [会議の主要議題]

## サマリー

[会議全体の要約を3～5行で記載]

## 決定事項まとめ

- [決定事項1]
- [決定事項2]

## TODOアイテム

- [TODO内容]（[担当]）

## 議題と討議内容

### 1. [議題名]

**討議内容**:

## 備考・補足事項

[追加で記録すべき情報があれば記載]
```

### 記載のポイント

#### 意思決定の明確化

- 「～することに決定」「～の方向で進める」などの決定事項は必ず記録
- 「～を検討する」「～を確認する」などの保留事項も明記
- クライアント様の承認・確認が必要な事項は特に強調

#### TODOアイテムの具体化

- 誰が（クライアント様 / クラスメソッド / ベンダー様）
- 何を（具体的なTODO内容）

を明確に記載

### 文字起こしからの抽出ポイント

- **議題の構造化**: 「次は～について」「～の件ですが」などから議題を抽出し、関連発言をグループ化
- **決定事項の抽出**: 「～でいきましょう」「～に決定」「了解です」「承知しました」などの表現
- **課題・懸念の抽出**: 「課題は～」「確認が必要」「どうしましょう」などの未解決事項
- **TODOの抽出**: 「～を確認します」「次回までに～」などのコミットメント

### リストの簡潔性

リストは一つの文章にまとめる意識を持つ。箇条書きの数が多く深い入れ子にならないように注意する。

悪い例:

```markdown
- SEC様より基本思想を説明いただきました
  - ネット切断時でも運営は継続必須です
  - ローカルネットワークでは動作継続します
```

良い例:

```markdown
- SEC様より基本思想を説明いただき、ネット切断時でもローカルネットワークで動作継続することを確認しました。
```

### 決定事項・TODOの簡潔性（蛇足を付けない）

決定事項・TODOは、決めた内容・やる内容を**1文で**書く。ステータスラベル（`（確定）`）・関連ID（`[[…]]`）・発言の引用（`根拠:「…」`）は本文に付けない。

決定事項の悪い例:

```markdown
- 三菱電機案件へのCortex導入は一旦保留とする。（確定・[[20260716-002]]の再確認）
- 日報・週次レポートをGold層に置く運用は見直す方向で認識が一致した。（保留）
```

決定事項の良い例:

```markdown
- 三菱電機案件へのCortex導入は一旦保留とする。
```

（未確定の「見直す方向」は決定事項まとめに載せず、「議題と討議内容」で触れる。`（確定）`は決定事項欄にあること自体が示すので不要。関連する過去決定は後段のGold層＝Decisionが `relations` で持つ）

TODOの悪い例:

```markdown
- お披露目会を来月中に1時間程度で開催する（高垣さん）。根拠:「どっかでお披め会はやりたくて…1時間ぐらい取ってもいいかもしれない」
```

TODOの良い例:

```markdown
- お披露目会を来月中に1時間程度で開催する（高垣さん）
```

（発言の逐語引用は照合の道具であって成果物ではない。担当は括弧で簡潔に。裏付けが曖昧な項目にだけ `⚠️要確認（理由）` を付ける）

### 注意事項

- 用語・人名は文字起こしの表記をそのまま転記せず、手順4の正規化マッピング（用語集の正式表記・名簿の正式表記）を適用する
- 技術的な専門用語は正確に記載
- 曖昧な表現は避け、具体的に記述
- 発言者の意図を正確に汲み取って要約
- クラスメソッド側の人は「さん」づけ。その他は「様」づけ
- `会議/` 内のこれまでの議事録から統一された書き方を参考にしてください

## トラブルシューティング

- **発言者が不明な場合**: 「発言者不明」と記載
- **聞き取れない部分**: 「[聞き取り不能]」と記載
- **議題が入り組んでいる場合**: 時系列ではなく議題ごとにまとめる
- **決定事項が曖昧な場合**: 「暫定決定」「要確認」などのステータスを明記

## 生成後の自己検証（第2パス）

議事録を書き終えたら、そのまま完了とせず、必ず以下の5チェックを1件ずつ実施してから確定する。ハルシネーション（実在しない課題キー・担当の取り違え・日付の矛盾等）はこの第2パスで検出する。

**共通原則**: **誤りの疑いは自動修正せず、該当箇所に `⚠️要確認` を付けて明示する**。正しい内容を上書きする過剰修正を防ぐため、修正ではなく指摘に留める。「AI自動生成・レビュー前」バナー（H1直後の1行）は自己検証後も**残す**（人間レビューの入口を消さない）。

1. **課題キーの実在チェック** — 議事録中の課題キー（例: `PROJ-123` 形式）を、課題ミラーディレクトリの実在キー一覧と突合する。取得例:

   ```bash
   grep -rhoE '課題キー: *[A-Z0-9_]+-[0-9]+' 課題管理/issues/ 2>/dev/null | grep -oE '[A-Z0-9_]+-[0-9]+' | sort -u
   ```

   （課題キーの表記・ミラーの配置は案件によって異なる。ファイル名から拾える案件はファイル名も併用する。）実在キー一覧に無いキーは**自動修正せず**、該当箇所に `⚠️要確認（実在しない課題キー）` を付ける。

2. **担当者の名簿突合** — 決定事項・TODO の担当者名を名簿（`Cortex/Members/records/` の `title`/`aliases`/`side`）と突合する。名簿に無い名前、および `side` の整合が怪しい割当（例: 顧客側が主導した意思決定を開発側メンバーの決定として記載している等）には `⚠️要確認` を付ける。名簿が無い案件はフォールバック（メンバー表ルール・CLAUDE.md の表）で突合する。

3. **日付×曜日の整合チェック** — 議事録中の「M/D（曜）」表記をすべて抽出し、実際の曜日と機械的に突合する。確認例:

   ```bash
   date -d "2026/07/07" +%a                    # Linux (GNU date。CIランナーはこちら)
   date -j -f "%Y/%m/%d" "2026/07/07" "+%a"    # macOS (BSD date)
   ```

   （`cal` でも可。）矛盾があれば**自動修正せず** `⚠️要確認（曜日不一致）` を付ける。

4. **決定/TODOのステータス規律** — 「〜する方向」「〜を依頼する」「持ち帰り」等の**未確定な発言は「決定事項まとめ」に格上げしない**（決定事項まとめは確定事項だけを載せる欄。保留・継続協議は「議題と討議内容」の中で触れる）。書き下すのは**決定・TODOそのものを1文で**。読み手が追加情報を必要としないよう簡潔に書く。
   - **蛇足を付けない**: `（確定）` `（保留）` 等のステータスラベル、`（[[20260716-003]]の運用継続）` 等の関連ID、`根拠:「〜」` の発言引用を**本文に書かない**。確定はこの欄にあること自体が示し、関連や出典は不要（必要なら後段のGold層＝Decisionが `relations` で持つ）。
   - **例外は要確認マークのみ**: 裏付けが取れない・曖昧な項目にだけ `⚠️要確認（理由）` を付ける。

5. **第2パス（突合検証）** — 執筆完了後、各決定・TODO・数値（金額・期日・件数等）を文字起こしと1件ずつ突き合わせる。**これは検証プロセスであって、引用を本文に残すためではない**（引用は照合の道具であり成果物ではない）。裏付けの取れない記述にだけ `⚠️要確認（文字起こしに根拠なし）` を付ける。
