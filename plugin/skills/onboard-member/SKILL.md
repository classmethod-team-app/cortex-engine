**既にセットアップ済みの案件コンテキストリポジトリに、新メンバーが参加してクローンした直後**に使うスキルです。(1) 参加者個人のローカル環境を整え、(2) 案件を理解する、までを支援します。

> `setup-project` との違い: `setup-project` は**リポジトリを新規作成する人**向け（リポジトリのSecrets設定など、1回だけ・全員に共通の準備）。本スキルは**既存リポに参加する人**向けで、リポ自体の準備は完了済みである前提。リポの準備状況は `/setup-status` で確認できる。

## パート1: ローカル環境の準備

上から順に確認・案内する（各ステップは実行前にユーザーへ確認）。**Node / pnpm / mise 等のランタイムは不要**（仕組みはすべて cortex プラグインと GitHub Actions が提供する）。

1. **GitHub 認証**: `gh auth status`。未認証なら `gh auth login`（クローン・PR・Issue 操作に使う）
2. **cortex プラグイン**: このリポジトリを Claude Code で開き、フォルダをトラストすると「cortex プラグインをインストールしますか」と自動で案内される。**「はい」を1回押すだけ**（ユーザー単位で1回きり。別案件のリポでも同じプラグインがそのまま使われる）。案件が職能ハーネス（PM/開発/デザイン等）を有効化している場合も、同じ案内でまとめて入る
3. **プラグインの自動更新トークン（推奨）**: マーケットプレイス（cortex-engine）は private リポのため、**起動時のバックグラウンド自動更新には認証トークンの環境変数が必要**（未設定でも `/plugin marketplace update` で手動更新は可能だが、改善が自動で届かなくなる）。チームの 1Password「環境」に共有トークンがあればそこから設定する。保存場所は動作環境で変わる → `credentials` ルール参照
4. **開発リポジトリ（submodule）**: 開発リポがある案件なら `git submodule update --init`（または `/clone-dev-repos`）
5. **Backlog の APIキー（`/backlog-push` を使う人だけ）**: まず「**エディタから課題への返信・Wiki更新を行いたいか**」を確認する
   - **読むだけの人は不要**: 課題ミラーは Webhook＋cron で自動同期されており、最新化は `/git-pull` で足りる。キーの配布自体が不要（クレデンシャルは持つ人が少ないほど安全）
   - **書き込みたい人のみ**: `DOMAIN`（共有値）/ `PROJECT_KEY`（共有値）/ `BACKLOG_API_KEY`（**各自の**APIキー）を環境変数で設定する。保存場所は動作環境で変わる（CLI＝`.env`／デスクトップ＝ローカル環境エディタ／Web＝クラウド環境設定）→ `credentials` ルール参照
   - **秘密情報はリポジトリにコミットしない**
6. **（任意）日本語ディレクトリの英語別名**: ターミナルで日本語ディレクトリ（`会議/` `共有資料/` 等）への移動が面倒な場合は、`sh "<SKILL_DIR>/scripts/setup-aliases.sh"` を実行すると英語別名（`Meetings` `Materials` 等）の symlink が張れる。**ローカルのみ・コミットされない**（正本は日本語のまま）。非エンジニアや別名が不要な人は実行しなくてよい。

## パート2: プロジェクトの理解（Gold起点のオリエンブリーフ）

`CLAUDE.md` の探索戦略（**Gold起点の二段構え**）に従い、生データの全文検索から始めず、まず `Cortex/`（Gold層）を読む。以下を読んで新メンバー向けのオリエンブリーフを作成する。

1. `Cortex/Home.md` — 案件の全体像・識別カード（kind / client / domains / platforms）・使用ツール
2. `Cortex/用語集/` — 案件固有の語彙（最初に揃えて表記揺れ・誤解釈を防ぐ）
3. `Cortex/Decisions/` — 主要な確定事項（特に現行版。`supersedes` で置き換えられた古い決定は現行で上書きされている点に注意）
4. `Cortex/レポート/` — 直近の動き・進行中の論点
5. 体制・関係者の資料（`CLAUDE.md` の案件固有ブロックや `Cortex/Home.md` にあれば）

### 出力するオリエンブリーフ

- **この案件は何か**: 目的・クライアント・業務ドメイン・プラットフォーム
- **体制と関係者**: 誰が何の役割か
- **直近の状況と進行中の論点**: （詳細は `/catch-up-recent-status` に委譲してよい）
- **押さえるべき主要な決定事項**（現行版）
- **重要な用語**（最小セット）
- **どこに何があるか**: `USAGE.md`（日々の使い方）・`README.md`（構成・コマンド一覧）へのポインタ
- **次の一歩**: よく使うコマンド（`/catch-up-recent-status`・`/post-meeting` 等）

## 注意事項

- 個人の認証情報は `.env` / 各自の環境に置き、**コミットしない**
- リポジトリ自体が未セットアップ（プレースホルダ `{{ }}` が残っている等）なら、まず `/setup-status`・`/setup-project` を案内する
- 旧方式（rulesync 時代）のリポジトリでは `.rulesync/` や `pnpm` の手順が README に残っていることがある。その場合はエンジン分離への移行（`/setup-status` 参照）を先に案内する
- 理解パートは必ず **Gold起点**（リポジトリ全体の全文検索から始めない）
