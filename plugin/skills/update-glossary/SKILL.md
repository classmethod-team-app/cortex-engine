案件固有の用語・定義を抽出し、`Cortex/Glossary/records/` 配下に **1用語1ファイル** で登録・更新します。スキーマ・運用ルールは `Cortex/Glossary/README.md` を参照。

## 目的

同じ概念が複数の表記で呼ばれている状態を放置すると、AIが解釈を誤る。用語集に定義と同義語を集約し、人とAIの語彙を揃える。

## 実行手順

### 1. 既存の用語集を読み込む

`Cortex/Glossary/records/*.md` のfrontmatterを読み、既存の `title` と `synonyms` の集合を把握する（重複登録・表記揺れの照合に使う）。

### 2. 情報源から用語候補を抽出する

`$ARGUMENTS` で情報源（ファイルパス・用語そのもの）が指定された場合はそれを優先する。指定がない場合は以下を走査する：

- `課題管理/documents/` 配下の用語集・要件定義書（既存の用語定義ドキュメント）
- `会議/` の議事録・`課題管理/issues/` の課題で頻出する案件固有の語
- `共有資料/` の提案書・仕様書

**抽出対象**: 案件・業界固有の用語、社内略語、一般語だがこの案件で特別な意味を持つ語、同一概念の表記揺れ、**案件で実際に使われる外部サービス・技術・規格で説明が要るもの**（例: Figma / LINEミニアプリ / OIDC 等。定義はステップ4でWeb一次情報から作り `references` を付ける）。
**除外**: ごく一般的で説明不要なIT基礎語（API、DB等）で、案件固有の意味も持たないもの。

### 3. 既存用語との照合

抽出した候補をステップ1の集合と照合する：

- **既存用語の表記揺れ** → 新規ファイルを作らず、既存ファイルの `synonyms` への追記を提案する
- **既存用語の定義変更** → 既存ファイルの本文更新を提案する（変更履歴はgitが持つ）
- **新規用語** → `template.md` ベースの新規ファイルを提案する

### 4. 定義の確定（一般公開用語はWeb検索で裏取り）

各用語を2種類に分け、定義の作り方を変える。

- **案件固有語・業界内部語**（その案件・社内でしか出てこない語）: 社内の情報源（議事録・Backlog・資料）の記述のみから定義する。**Web検索しない**（公開情報が無い／案件文脈が正）。`references` は付けない。
- **一般に公開されているサービス・技術・標準語**（例: Figma / LINEミニアプリ / OIDC / Backlog 等の外部サービス・技術・規格）: **WebSearch / WebFetch で公式・一次情報を確認**してから定義する。推測で書かない。
  - 公式ドキュメント・提供元サイト等の**一次ソースを優先**（個人ブログ等の二次情報は避ける）
  - 確認した一次ソースのURLを `references` に1〜数件記載する
  - 一般的な意味を書いた上で、**この案件での使われ方が特殊なら本文でその点も補足**する（例: 「本案件ではFigmaの〜機能のみ使用」）
  - 検索結果が古い・矛盾する・確証が持てない場合は、その旨をユーザーに伝え、断定しない

判別に迷う語（一般語に見えるが案件で特別な意味、等）は、両方の観点で書く（一般定義＋案件での意味）。

### 5. プレビューと承認（必須）

登録・更新する内容（新規／synonyms追記／定義更新の別、ファイル名、定義文、`source`、`references`）を提示し、ユーザーの承認を得てから書き込む。**承認なしに書き込まない。** 一般公開用語については `references` のリンクも提示する。

### 6. ファイルの作成・更新

- ファイル名: `records/{代表表記}.md`、ID: `term:{代表表記}`
- `source` に**この案件での出典**（BacklogドキュメントURL・議事録の安定ID等）を記載する
- 一般公開用語は `references` に**一次情報のURL**を記載する（案件固有語は省略）
- 使われなくなった用語は削除せず `status: superseded` に変更する

## 注意事項

- 同一概念に複数ファイルを作らない（表記揺れは `synonyms` に集約）
- 出典となるBronzeのドキュメント（`課題管理/documents/` 等）は読むだけで編集しない
- 定義を推測で書かない。情報源に根拠がある定義のみ登録し、根拠が弱い場合はその旨をユーザーに伝える
