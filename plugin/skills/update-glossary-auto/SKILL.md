`/update-glossary`（手動・承認あり）の**自律実行版**です。前回実行以降に追加・更新されたコンテキストから新規用語を抽出し、`Cortex/用語集/` へ直接コミットします（push はワークフロー側の責務）。

承認は**事後レビュー方式**です。AIが生成した用語は必ず `status: draft` を付け、人間が定義を確認・修正して `status: active` に変えることでレビュー完了とします。Viewer は draft の用語に「AI生成・未確認」の注意を表示します。事前承認（PRマージ）ではなく事後レビューにするのは、毎晩のPR運用がスケールしないためです。その代わり、自動化の範囲を**新規追加と synonyms 追記のみ**に限定し、既存定義のドリフトを構造的に防ぎます。

## 設計の要点

- **AI生成は必ず `status: draft`**: 人間がレビューするまで `active` にしない。draft はViewerで「未確認」として明示され、人間が定義を確認・置き換えてから `active` に昇格させる
- **新規追加と synonyms 追記のみ**: 既存用語の定義本文・`title`・`id`・`status` は自動で書き換えない。定義の変更が必要と判断した場合はファイルを変更せず、コミットメッセージまたは実行サマリに提案として記載するに留める
- **増分方式**: 直近約25時間のgit差分のみを走査する。新規候補ゼロは「コミットなし」で正常終了（毎晩走るため通常の状態）
- **除外リストの尊重**: `Cortex/用語集/README.md` の「除外用語」セクションに列挙された語は追加しない（過去にレビューで却下された語の再追加防止）

## 実行手順

### ステップ 1: 差分ソースの特定

**前回成功実行以降**（環境変数 `SINCE`。ワークフローが「直近の成功run時刻」を渡す＝失敗が続いても次の成功時に失敗分まで遡って巻き取る）に追加・更新された、用語が含まれうるファイルを対象にする。`SINCE` が無い手元実行では約 25 時間をデフォルトにする。`Cortex/用語集/` 配下は必ず除外する（自己参照防止）。

```bash
git log --since="${SINCE:-25 hours ago}" --name-only --pretty=format: -- \
  課題管理/issues/ 課題管理/documents/ 会議/ 共有資料/ Cortex/Decisions/ \
  | sort -u | grep -E '\.md$' || true
```

対象が0件なら、以降をスキップして「追記なし」で正常終了する。

### ステップ 2: 既存用語・除外リストの読み込み

> ⚠️ **既存用語を 1 ファイルずつ開いて全文 Read しない**。用語は日付スコープがなく全レコードが照合対象になるが、必要なのは frontmatter の `title`/`synonyms` だけで、**grep 1 コマンドで全件分を取得できる**。件数が増えても定数コストで済み、無人実行のターン上限を食い潰さない。

```bash
# 既存の title / synonyms を全レコードから一括取得（全文 Read しない・draft 含む全件が照合対象）
grep -hE '^(title|synonyms):' Cortex/用語集/records/*.md 2>/dev/null || true
```

- 上記で既存の `title` と `synonyms` の集合を把握する（draft を含む全レコードが照合対象）
- `Cortex/用語集/README.md` の「除外用語」セクションに列挙された語を把握する

### ステップ 3: 新規用語候補の抽出

ステップ1の対象ソースから候補を抽出し、以下で絞り込む：

- **対象**: 案件・業界固有の用語、社内略語、一般語だがこの案件で特別な意味を持つ語、案件で使われる外部サービス・技術・規格で説明が要るもの
- **案件固有語**は、定義が**情報源に明示されている語のみ**登録する。文脈から推測した定義は書かない。`references` は付けない
- **一般公開のサービス・技術・標準語**は、**WebSearch / WebFetch が利用可能なら公式・一次情報を確認**して定義し、一次ソースのURLを `references` に記載する。**Webツールが使えない実行環境では、その語の定義は書かず候補に留める**（無理に推測で書かない。人間のdraftレビュー時に定義＋リンクを補う想定）
- 既存の `title` / `synonyms` と一致または実質同義 → 新規ファイルを作らず、該当ファイルへの **synonyms追記** とする
- 除外リストに該当 → スキップ
- 確信が持てない語は登録しない（過剰登録はノイズとなり用語集の信頼を損なう。直コミットされるため、ローリングPR方式よりさらに保守的に判断する）

### ステップ 4: ファイル作成・更新とコミット

- 新規用語: `Cortex/用語集/template.md` をベースに `records/{代表表記}.md` を作成（スキーマは `Cortex/用語集/README.md`）。**`status` は必ず `draft`**。`source` に案件内の出典（安定IDまたはURL）を記載する。一般公開用語でWeb一次情報を確認できた場合は `references` にそのURLを記載する（案件固有語は省略）
- synonyms追記: 該当ファイルのfrontmatterの `synonyms` のみを変更する（`status` は触らない）
- **コミット前に `node "<SKILL_DIR>/../../scripts/validate-cortex.mjs"`（プラグイン同梱リンター） を実行し、規約違反があれば修正する**（違反したままコミットしない）
- 変更があれば `git commit` までを行う（push は行わない）。コミットメッセージは日本語で簡潔に、AI署名は付けない
- 候補が0件なら何もコミットせず正常終了する

## 注意事項

- 既存用語の定義本文・`title`・`id`・`status` は変更しない（draft → active への昇格は人間のレビュー作業）
- READMEの「除外用語」セクションを編集しない（編集は人間のレビュー時のみ）
- Bronzeのドキュメント（`課題管理/` 等）は読むだけで編集しない
- 既存用語の照合は **frontmatter の grep** で行い、**1 ファイルずつ全文 Read しない**（件数増でターン超過するため。ステップ 2 参照）
