## このリポジトリの動かし方（エンジン分離アーキテクチャ）

本リポジトリは**データ（コンテキスト）と薄い設定だけ**を持ちます。スキル・ワークフロー・スクリプトなどの仕組みは中央リポジトリ **cortex-engine** で版管理され、Claude Code プラグイン（`.claude/settings.json` 参照）と reusable workflows（`.github/workflows/` のスタブ）経由で配布されます。仕組みの改善はエンジン側で行い、このリポジトリでは行いません。

## データ階層（Bronze / Silver / Gold）

データは精製度で3層に捉えます。

- **Bronze（生記録）**: 文字起こし・課題のやり取り（`課題管理/`）・資料原本・ソースコード。外部サービスの**同期ミラーは手編集しない**（修正は正本側で行い再同期する）
- **Silver（中間成果物）**: 議事録（`会議/`）・ドキュメント・Wiki・変換済み資料（`共有資料/`）
- **Gold（確定の判断材料）**: `Cortex/` 配下（Decisions・用語集・レポート・Home）。AIが最初に読む入口

## 作業開始時の探索戦略: Gold起点の二段構え

**推測で作業せず、必ず関連コンテキストを探索してください。** 最初から全文検索で生データを総当たりしない。

1. **第一段: `Cortex/`（Gold層）を読む** — `Cortex/Home.md`（全体像）→ `Cortex/用語集/`（語彙）→ `Cortex/Decisions/`（確定事項）→ `Cortex/レポート/`（直近の動き）
2. **第二段: 関係を辿って必要な生データだけ開く** — Gold層のfrontmatter（`relations`/`source`/`references`）は安定ID（課題キー・`minute:{定例名}:{日付}` 等）で出典を指す。経緯・一次情報が必要なときだけ、その指す先（`課題管理/issues/`・`会議/` 等）をピンポイントで読む
3. **フォールバック**: Goldから辿れない情報（未精製の直近のやり取り・実装詳細）だけ、生データ側をGrep等で直接探索する

frontmatterの規約（エンティティ型・ID規則・リレーション）の正本は cortex-engine の `docs/ontology.md`。

## 公開範囲と内部情報の扱い

本リポジトリは顧客が生でアクセスする前提（AIS）で運用しうるため、**「このリポジトリに置いたもの＝顧客に見えるもの」**とみなすこと。内部限定情報（売上・利益率・原価・アサイン工数・単価・人事・率直なベンダー評価等）は書き込まない（AIも生成・書き出しをしない）。※`kind: 社内プロジェクト` の案件は対象外だが、原則として同じ規律で運用する。内部情報を手元で併用したい場合は private リポの submodule として取り込む（顧客はアクセス権を持たないため見えない）。

## 運用原則（要点）

- 同期ミラー（`課題管理/`・`デザイン/inventory/` 等）は手編集しない
- frontmatterは生成者（スキル・エクスポーター）が付与する。関係は安定IDで張る（ファイルパス参照禁止）
- Gold層の自動生成は追記のみ・draftマーキング・push前スキーマ検証（人間のレビューが届く形にする）
- `Cortex/Home.md` は Viewer の入口（顧客・部外者が読む面）。**エンジン・ハーネス等「仕組み」への参照は書かない**（仕組みの参照は本ファイルと README が持つ）
